# 기여 가이드라인 (Contributing Guide)

Livith-Skills 저장소에 기여해 주셔서 감사합니다! 이 가이드라인은 원활한 협업을 위해 꼭 읽어주세요.

---

## 🏗️ 저장소 구조

스킬은 다음과 같은 구조로 추가해주세요:

```
skills/
└── {skill-name}/
    ├── README.md      # 사람을 위한 스킬 설명서
    └── skill.md       # 실제 에이전트 프롬프트 (에이전트용)
```

### 파일 설명

- **README.md**: 스킬의 목적, 사용법, 작동 원리, 주의사항 등을 사람이 읽을 수 있도록 작성합니다.
- **skill.md**: 실제 에이전트에게 전달할 프롬프트/지시사항을 작성합니다. 이 파일이 곧 에이전트가 사용할 스킬 자체입니다.

---

## ✍️ 스킬 작성 시 참고사항

### 스킬 이름
- 영문 소문자와 하이픈(-)을 사용합니다.
- 예: `code-review`, `doc-generate`, `commit-message`

### README.md 에 포함하면 좋은 내용
1. 스킬의 목적과 해결하고자 하는 문제
2. 사용 대상 에이전트 (예: Claude, GPT-4, Copilot 등)
3. 사용 방법 및 예시
4. 주의사항이나 제약사항
5. 버전 정보 (필요한 경우)

### skill.md 작성 팁
- 명확하고 구체적인 지시사항을 작성합니다.
- 에이전트의 역할을 명확히 정의합니다.
- 출력 형식이나 제약사항을 구체적으로 명시합니다.
- 보안상 민감한 정보(API 키, 비밀번호, 내부 URL 등)는 절대 포함하지 마세요.

## 📝 커밋 메시지 규칙

커밋 메시지는 다음 양식을 따릅니다:

### 이슈 번호가 있는 경우
```
type/issue-number: message
```
**예시:** `feat/#15: add code-review skill`

### 이슈 번호가 없는 경우
```
type: message
```
**예시:** `chore: update gitignore`

### Type 목록

| Type | 설명 |
|------|------|
| `feat` | 새로운 스킬 추가 |
| `fix` | 기존 스킬 버그 수정 |
| `docs` | README, 문서 수정 |
| `chore` | 저장소 설정, 템플릿 변경 |
| `update` | 스킬 내용 업데이트/개선 |
| `remove` | 스킬 삭제 |

---

## 🔄 Pull Request 절차

1. 새로운 브랜치를 생성합니다.
2. 스킬을 추가/수정/삭제합니다.
3. PR 템플릿을 작성합니다.
4. 최소 1개 이상의 에이전트에서 테스트한 결과를 포함합니다.
5. 리뷰를 요청합니다.
